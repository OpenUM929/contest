## 1. 배경 및 목적

현재 `이음(以音)`은 공공데이터 포털로부터 이미지/음원/텍스트를 수집하고, 복지사가 이 중 하나를 선택하여 이번 주 대화 주제를 발행합니다.

이 과정에서 **이미지를 선택할 때** 복지사가 단순히 "옛 사진"이라고만 보고 넘어가는 경우가 많습니다. 이미지 속 장면의 시대적 맥락, 감성적 분위기, 그리고 그것이 노인/청년에게 어떤 이야기를 불러일으킬 수 있는지를 미리 파악하면 복지사의 주제 선택이 훨씬 풍부해집니다.

따라서 **국립청주박물관 소장품 메타데이터(유물명, 설명, 재질, 규격)**를 입력으로 받아, 복지사가 주제를 선택할 수 있도록 돕는 **이미지 분석 프롬프트**가 필요합니다.

---

## 2. 기존 철학의 적용 한계

설문지 프롬프트에서 정립한 "안개비" 철학은 **사용자(노인/청년)가 대화하는 순간**에 적용되는 것입니다.

그러나 이미지 선택은 **복지사가 주제를 고르는 단계**입니다. 여기서 안개비 효과를 억지로 노리면:
- 이미지 분석 결과가 과도하게 감성적이고 추상적이 됩니다
- 복지사에게 "이 이미지는 외로움을 치유합니다" 같은 해석을 강요하게 됩니다
- 이미지의 객관적 정보(시대, 장소, 사건)가 묻힙니다

**따라서 이미지 선택 프롬프트는 안개비 철학을 직접 적용하지 않습니다.** 대신, **정보 제공 + 감성적 힌트**를 균형 있게 섞어 복지사가 스스로 판단할 수 있도록 돕습니다.

---

## 3. 설계 원칙

### 3-1. 정보의 균형

| 유형 | 비율 | 설명 |
|------|------|------|
| **지식적 정보** | 40% | 유물의 시대, 종류, 재질, 용도 등 객관적 사실 |
| **감성적 정보** | 30% | 유물이 주는 분위기, 연상되는 것, 느낌 (억지 금지) |
| **대화 주제 후보** | 30% | 이 유물을 기반으로 할 수 있는 2~3개의 주제 방향 |

### 3-2. 금지사항

- 이미지가 "외로움을 치유한다"는 식의 해석 강요
- "이 유물은 노인들에게 이런 감정을 불러일으킵니다" 같은 감정 지정
- 메타데이터에 없는 가상의 서사를 지어내는 것
- 단순히 "예쁜 유물", "옛날 느낌" 등 추상적 표현만 나열

### 3-3. 허용사항

- "이 유물은 조선시대 청자로, 재질은 도자기이며 주로 제례에 사용되었습니다" (지식)
- "이 유물은 단아하고 고요한 분위기를 줍니다" (감성)
- "이 유물을 주제로 하면 '어머니의 그릇', '제사 준비하던 날' 등으로 이야기를 이어갈 수 있습니다" (후보)

## 4. 제안 프롬프트 구조

> **입력**: 청주박물관 API가 반환하는 텍스트 메타데이터 (`TITLE`, `DESCRIPTION`, `INGREDIENT`, `SIZING`)
> **모델**: DeepSeek 또는 OpenCode (Vision 불필요 — 텍스트 전용)

```
[역할]
당신은 '이음(以音)'의 유물 분석 도우미입니다.
복지사가 국립청주박물관 소장품을 보고 대화 주제를 선택할 수 있도록,
유물의 메타데이터를 바탕으로 정보를 균형 있게 정리해 주세요.

[입력 정보]
- 유물명: {title}
- 유물 설명: {description}
- 재질: {ingredient}
- 규격: {sizing}
- 소장기관: {source}

[출력 요구사항]
1. 유물 요약 (객관적 사실)
   - 시대/연도 추정 (메타데이터에서 파악, 불명확하면 "추정" 표시)
   - 유물 종류 (도자기/회화/생활용품/의복 등)
   - 주요 특징 (재질, 용도, 형태)

2. 지식적 맥락 (선택적)
   - 이 유물이 갖는 역사적/사회적 배경
   - 이 시대에 이 유물이 어떤 의미였는지
   - 단, 메타데이터에 없는 정보는 "추정"으로 표시

3. 감성적 힌트 (억지 금지)
   - 이 유물이 주는 분위기나 느낌
   - 이 유물을 보면 연상되는 장면, 소리, 냄새 (가상 서사 지어내기 금지)

4. 대화 주제 후보 (2~3개)
   - 이 유물을 기반으로 할 수 있는 대화 주제 방향
   - 각 후보에 "이 주제는 ~에 대한 이야기를 나눌 수 있습니다"라는 한 줄 설명

5. 주의사항
   - 메타데이터에 없는 인물이나 사건을 지어내지 마세요
   - 유물이 "무조건 외로움/상처를 치유한다"는 식의 해석을 강요하지 마세요
   - 복지사는 이 정보를 참고하여 스스로 주제를 선택합니다
   - 모든 출력은 한국어로 작성하세요

[출력 형식]
JSON 형식으로만 출력 (다른 텍스트 없이):
{
  "artifact_summary": {
    "era": "시대 (불명확 시 '추정' 포함)",
    "type": "유물 종류",
    "features": ["주요 특징1", "주요 특징2"]
  },
  "context": {
    "historical": "역사적 맥락 (없으면 null)",
    "social": "사회적 맥락 (없으면 null)"
  },
  "mood": {
    "atmosphere": "분위기",
    "associations": ["연상1", "연상2"]
  },
  "topic_candidates": [
    {
      "title": "주제 후보 제목",
      "description": "한 줄 설명",
      "age_suitability": "elderly/youth/both"
    }
  ]
}
```

## 5. 구현 방식

### 5-1. 백엔드 연동

- `fetch_cheongju_museum()`이 반환하는 후보 dict에는 이미 `title`, `description`, `ingredient`, `sizing`이 포함되어 있음
- 복지사가 이미지 카드를 클릭할 때 `POST /api/welfare/topics/analyze` 호출 (lazy loading)
  - 검색 시점에서 분석하면 후보 5개 × AI 호출 = 응답 지연 발생
- AI 모델: DeepSeek (`DEEPSEEK_API_KEY`) 또는 OpenCode (`OPENCODE_API_KEY`) — Vision 불필요
- 분석 결과 JSON을 응답으로 반환, 프론트엔드에서 카드 상세 패널에 표시
- JSON 파싱 실패 시 폴백:
  ```python
  try:
      result = json.loads(raw_output)
  except json.JSONDecodeError:
      result = {
          "artifact_summary": {"era": "분석 불가", "type": "", "features": []},
          "context": {"historical": None, "social": None},
          "mood": {"atmosphere": "", "associations": []},
          "topic_candidates": []
      }
  ```

### 5-2. 복지사 대시보드 UI

```
[이미지 카드]
┌─────────────────────────┐
│  [이미지 썸네일]         │
│                         │
│  🏺 조선시대, 도자기      │
│  단아하고 고요한 분위기   │
│                         │
│  [분석 보기]             │
└─────────────────────────┘
        ↓ 클릭 (lazy load)
┌─────────────────────────┐
│  유물 분석               │
│  • 시대: 조선시대 추정    │
│  • 종류: 청자 도자기      │
│  • 분위기: 단아, 고요함   │
│  • 주제 후보:            │
│    - 어머니의 그릇        │
│    - 제사 준비하던 날     │
│  [이 주제로 발행하기]     │
└─────────────────────────┘
```

## 6. 결론

이미지 선택 프롬프트는 **설문지 대화 프롬프트와 다른 목적**을 가집니다. 설문지는 "사용자의 마음을 어루만지는 질문"이 목적이라면, 이미지 선택은 **"복지사가 현장의 사람들에게 맞는 주제를 고르는 데 필요한 정보를 제공"**하는 것이 목적입니다.

따라서:
- 안개비 철학을 **억지로 적용하지 않음**
- 단순 지식적 정보와 개인적 감성 표현을 **균형 있게 허용**
- **Vision API 불필요** — 청주박물관 메타데이터(텍스트)로 충분
- 분석은 **복지사가 카드를 클릭할 때** lazy loading (검색 시 지연 방지)
- JSON 파싱 실패 대비 폴백 처리 필수
- 복지사가 **스스로 판단**할 수 있도록 돕는 것에 초점

이 설계를 바탕으로 다음 단계에서 실제 프롬프트 파일을 작성하고, 백엔드/프론트엔드 연동을 구현합니다.
