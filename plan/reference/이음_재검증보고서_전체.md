# 이음(以音) — 기획서 대비 실제 코드 재검증 보고서

> **작성 목적**: 『이음_상세계획서.md』의 주장이 실제 `ium/` 코드에서 어느 수준까지 구현되었는지 냉정하게 대조
> **검증 기준일**: 2026년 6월 1일
> **검증 대상**: `C:\dev\contest\ium\` 전체 코드베이스
> **평가 원칙**: "코드가 존재한다" ≠ "구현되었다". 실제 작동 여부와 기획서 주장의 일치도를 기준으로 판단

---

## 1. 검증 방법

1. 기획서의 **모든 주장(텍스트)**을 항목별로 추출
2. 해당 주장을 뒷받침하는 **실제 코드**를 파일·함수 단위로 추적
3. **실제 호출 흐름**을 따라가며, 함수가 정의만 되어 있는지 / 실제로 호출되어 작동하는지 구분
4. 기획서 표현("구현", "연동", "완료", "활용")과 코드의 실제 상태를 대조

## 3. 심각한 코드 품질 문제 및 버그

### 버그 1: 수필 생성 시 주제 미필터링 (Critical)
- **위치**: `backend/app/routers/essay.py` 24~29라인
- **내용**: `select(Conversation.content).where(Conversation.role == "user")` — `topic_id` 조건이 없어 **모든 주제의 대화가 수필에 섞임**
- **영향**: 이번 주 주제와 무관한 과거 대화가 수필에 포함되어 품질 저하

### 버그 2: AI 모델명 오류 (High)
- **위치**: `backend/app/services/claude.py` 66라인
- **내용**: `model="claude-sonnet-4-6"` — Anthropic 공식 모델명은 `claude-3-5-sonnet-20240620` 등이며, `"claude-sonnet-4-6"`는 유효하지 않을 가능성이 매우 높음
- **영향**: 실제 API 호출 시 404 오류 발생 가능. PoC 데모가 깨질 수 있음

### 버그 3: 로컬호스트 하드코딩 (High)
- **위치**: `ElderlyMain.tsx` 10라인, `YouthMain.tsx` 10라인, `Dashboard.tsx` 4라인
- **내용**: `const API = "http://localhost:8000"` 하드코딩
- **영향**: 실제 기기(모바일)에서는 localhost가 PC를 가리키지 않아 API 호출 불가. 데모 시 반드시 수정 필요

### 버그 4: Dashboard resolveAlert dummy 함수 (Medium)
- **위치**: `dashboard/src/pages/Dashboard.tsx` 41~44라인
- **내용**: `resolveAlert`이 실제로는 `alert_id` 없이 `fetchDashboard()`만 호출. "해결 처리"가 되지 않음
- **영향**: 복지사가 UI에서 "해결" 버튼을 눌러도 아무 일도 일어나지 않음

### 버그 5: 주제 자동 갱신 로직 없음 (Medium)
- **위치**: `public_data.py` 51~60라인
- **내용**: `get_active_topic()`이 DB를 조회하지 않음. `weekly_topics` 테이블은 `init.sql`에서 생성되나, 주제를 DB에 삽입하는 로직이 전무
- **영향**: 주제가 하드코딩된 샘플 5개로 영원히 고정. 새로운 주제 추가 불가
