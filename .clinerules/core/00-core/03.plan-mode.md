# 📌 Plan Mode 규칙

## Plan Mode이란?

- Plan mode에서는 **읽기만**
- 실행은 "수행"이라고 **명시적으로 요청**할 때까지 대기

## 계획서 작성 규칙

Plan Mode에서 사용자가 계획서(또는 설계 문서, 상세 계획) 작성을 요청한 경우:

1. **저장 위치**: 프로젝트 루트 기준 `plan/` 폴더 아래, **작업별 하위 폴더**를 생성하여 저장한다.
   - 작업 폴더명: `MMDD_NN_작업명/`
   - 모든 계획 문서는 해당 작업 폴더 내부에 저장
   - 예: `plan/0605_03_deploy-retry-cascade/`

2. **파일명 형식**: `MMDD_NN_작업명.md`
   - `MMDD`: 작성일 월일 (예: `0605`)
   - `NN`: 당일 동일 작업의 순번 (01, 02, ...)
   - `작업명`: 하이픈(-)으로 연결된 영문 소문자, **최대 20자**
   - 예: `0605_03_deploy-retry-cascade.md`

3. **파일 내용**: 반드시 마크다운 형식. 최상단에 제목, 작성일시, 작업 유형, 상태 기재

4. **승인 절차**: 사용자가 "수행"으로 명시적 요청 전까지는 파일 시스템에 **저장만** 하고 실제 코드 변경은 하지 않음

5. **사용자 알림**: 계획서를 사용자에게 안내할 때, 대화창 마지막에 반드시 **파일 전체 경로**와 **파일명**을 함께 출력
   - 예: `계획서 저장 위치: plan/0605_03_deploy-retry-cascade/0605_03_deploy-retry-cascade.md`

6. **상태 관리**: 계획서 상태는 **단일 파일 내 인라인으로 관리**한다. 별도 복사본 파일을 생성하지 않는다.
   - 상태 약어:
     - `PND`: Pending (초안/승인 대기)
     - `DN`: Done (작업 완료)
   - 상태는 문서 최상단 메타 블록에 한 줄로 표기:
     - 예: `> 상태: PND | 작성일: 2026-06-05`
     - 예: `> 상태: DN | 완료일: 2026-06-05`

7. **수정 이력 관리 (인라인 히스토리)**: 계획서에 수정/재검토 사항이 발생한 경우 별도 파일 생성 없이 단일 파일 내에서 관리한다.
   - 문서 최상단(메타 블록 바로 아래)에 `## 수정 이력` 섹션을 두고 **요약 테이블**로만 기록한다.
   - 이전 수정 요청의 상세 내용은 요약 테이블 한 줄로 압축하고, 문서 내 해당 섹션을 최신 내용으로 직접 교체한다 (이전 전체 내용 복사 금지).
   - 현재 수정 요청 상세 내용은 문서 내 별도 섹션으로 전체 기록한다.
   - 완료 시 수정 이력 섹션은 그대로 보존하고 상태만 `DN`으로 변경한다.

   ```markdown
   ## 수정 이력

   | 날짜 | 변경 섹션 | 변경 요약 |
   |------|-----------|-----------|
   | 2026-06-05 | §3 구현 방식 | retry API 엔드포인트 경로 변경 |
   | 2026-06-05 | §5 결함 목록 | 결함 2 (sessionId 누락) 추가 |
   ```

8. **작업 디렉토리 내 테스트/결과 관리**: 계획서 기반으로 구현 시 테스트가 필요한 경우, 해당 작업 폴더 안에 `test/` 하위 폴더를 생성하여 테스트 코드 파일을 저장한다. 테스트 완료 보고서 및 리포트는 동일 폴더 안에 `result/` 하위 폴더를 생성하여 저장한다.
   - 예시 구조:
      ```
       plan/
       └── 0605_03_deploy-retry-cascade/
           ├── 0605_03_deploy-retry-cascade.md
           ├── test/
           │   └── test_retry_cascade.py
           └── result/
               └── 완료보고_260605-1500.md
      ```

## Git 커밋/푸시 시 `.clinerules` 지침 파일 취급 규칙

> ⚠️ **협업 보존 규칙**: 다른 팀원이 `.clinerules`를 동시에 작업할 수 있으므로, 반드시 아래 규칙을 준수한다.

1. **교체 금지**: `.clinerules/` 하위 파일을 커밋할 때, **파일 전체를 교체(replace/overwrite)**하지 않는다.
2. **차이 확인**: 푸시 전 `git diff HEAD~1 -- .clinerules/` 또는 `git show origin/master:.clinerules/파일명`으로 원격 내용과 비교한다.
3. **추가 방식**: 기존 내용이 유지된 상태에서, **우리가 추가한 항목만 Append** 하는 방식으로 커밋한다.
4. **삭제 방지**: 기존 지침의 섹션, 규칙, 문맥이 **삭제되거나 누락되지 않도록**, `git diff --cached -- .clinerules/`로 변경 내역을 반드시 검토한다.
5. **충돌 대응**: 원격 파일과 로컬 파일이 다를 경우, `git merge` 방식으로 양쪽 내용을 병합한 후 커밋한다. 병합 불가 시 사용자에게 보고한다.
