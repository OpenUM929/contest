# 이음(以音) — 복지사 중심 설문지(선택형/서술형) 통합 관리 시스템 구현 계획 (확정안)

> **작성 기준**: 2026년 6월 2일  
> **참고 문서**: `doc/주제발행_4단계_지문.md`, `이음_구현계획_A_주제선정_복지사중심.md`  
> **결정 사항 반영**:
> 1. **혼합형(mixed)**: 단순 순차 흐름 (`choice` → `narrative` 단방향, 분기 없음)
> 2. **UI 구조**: 별도 `설문지 통합 관리` 메뉴 신규 (주제 발행과 설문지 관리 분리)
> 3. **선택형 보기**: 복지사가 개수까지 **자유롭게 추가/삭제/순서 변경** 가능

> **mixed 단순 순차 규칙**: `questions` 배열 내 `type: "choice"`가 **최대 1개, 반드시 첫 번째**에 위치. 이후 모든 항목은 `type: "narrative"`. 분기(`next_map`) 없음. 사용자가 choice를 선택하면 즉시 다음 narrative 질문이 노출됨.

### 2-2. Schema 규칙 요약 (확정)

| 필드 | 설명 | 필수 | 비고 |
|------|------|------|------|
| `schema_version` | "1.0" | O | |
| `question_type` | "choice" \| "narrative" \| "mixed" | O | |
| `questions` | 질문 객체 배열 | O | `mixed`는 2개 이상, 첫 항목은 반드시 `choice` |
| `questions[].id` | "q1", "q2"... | O | |

| `guidelines` | 답변 가이드라인 | X | |
| `placeholder` | 입력 힌트 | X | |
| `suggested_duration_seconds` | 서술형 권장 녹음 시간(초) | X | narrative 전용 |

## 3. AI 프롬프트 설계 (4단계 지문 기반)

> **핵심 설계 원칙**: 프롬프트 텍스트로 JSON 출력을 요청하는 방식은 사용하지 않는다.  
> 대신 **`instructor` 라이브러리 + Pydantic 모델**을 통해 스키마를 API 레벨에서 강제한다.  
> 프롬프트 파일은 AI에게 "무엇을 만들지"만 지시하고, "어떤 형식으로 출력할지"는 instructor가 담당한다.

### 3-1. 프롬프트 파일 위치

```
backend/app/prompts/
├── topic_publish_choice_v1.txt
├── topic_publish_narrative_v1.txt
└── topic_publish_mixed_v1.txt
```

> `topic_publish_schema_v1.json` 제거 — Pydantic 모델(`QuestionSet`)이 단일 진실 공급원이므로 별도 JSON 파일 불필요.

### 3-2. 선택형 (choice) 프롬프트

> **`[출력 JSON]` 섹션 없음** — instructor가 `QuestionSet` 스키마를 tool_use로 AI에게 자동 전달하므로 프롬프트에서 출력 형식을 반복 기술할 필요 없음.

```text
[역할]
당신은 세대공감 플랫폼 '이음(以音)'의 대화 질문 설계 AI입니다.
복지사가 사용자(노인/청년)에게 던질 '선택형' 대화 주제와 질문을 설계하세요.

[입력 정보]
- 주제 제목: {title}
- 주제 설명: {description}
- 미디어 타입: {media_type}
- 대상 연령대: {target_age}
- 복지사 힌트(있으면): {custom_hint}

[지시사항]
1. 주제와 미디어를 바탕으로, 대상 연령대가 자연스럽게 대화를 시작할 수 있는 질문을 설계하세요.
2. 보기(options)는 4~5개 생성하세요.
3. 반드시 마지막 보기는 "기타 (직접 말씀해 주세요)" 형태(is_other: true)로 포함하세요.
4. 노인용일 경우 보기 문구는 공백 포함 15자 이내, 친근하고 구어체로 작성하세요.
   예) "채소·생선 좌판" (8자), "따뜻한 분식집" (8자)
5. 각 보기에 어울리는 icon_hint 이모지를 1개씩 포함하세요.
6. guidelines를 2~3개 작성하여 사용자의 답변 방향을 안내하세요.
```

### 3-3. 서술형 (narrative) 프롬프트

```text
[역할]
'이음(以音)' 대화 질문 설계 AI입니다. '서술형' 질문을 설계하세요.

[입력 정보]
- 주제 제목: {title}
- 주제 설명: {description}
- 미디어 타입: {media_type}
- 대상 연령대: {target_age}

[지시사항]
1. 사용자가 자유롭게 이야기할 수 있도록 유도하는 열린 질문을 만드세요.
2. guidelines를 2~4개 포함하여 답변 방향을 안내하세요.
3. placeholder를 제공하세요 (예: "그날의 이야기를 들려주세요...").
4. 노인용일 경우 suggested_duration_seconds를 30~60 사이로 설정하세요.
```

### 3-4. 혼합형 (mixed) 프롬프트 — 단순 순차

```text
[역할]
'이음(以音)' 대화 질문 설계 AI입니다. '혼합형(선택형→서술형)' 질문을 설계하세요.
선택형은 1개만, 이후 서술형이 1개 이상 이어집니다. 분기(branching)는 없습니다.

[입력 정보]
- 주제 제목: {title}
- 주제 설명: {description}
- 미디어 타입: {media_type}
- 대상 연령대: {target_age}

[지시사항]
1. 첫 질문은 반드시 'choice' 타입입니다 (진입 질문).
   - 보기는 3~4개, 마지막에 기타(is_other) 포함.
   - 노인용 보기 문구는 공백 포함 15자 이내.
2. 두 번째 질문은 반드시 'narrative' 타입입니다.
   - choice에서 어떤 보기를 골랐든 동일한 narrative 질문으로 이어집니다.
3. 필요하면 세 번째 narrative 질문도 추가할 수 있습니다.
```

## 4. 백엔드 구현 설계

### 4-1. `question_parser.py` — Pydantic 스키마 모델 (instructor 연동)

> **설계 변경**: `extract_json_from_ai_response()`, `parse_and_validate_questions()` 함수 제거.  
> JSON 추출·파싱·재시도는 모두 `instructor`가 담당한다.  
> 이 파일은 **Pydantic 모델 정의만** 담당하며, 백엔드 전체에서 단일 스키마 공급원으로 사용된다.

```python
# backend/app/services/question_parser.py (신규 파일)
# Python 3.10+ 필요 (Union 대신 | 문법 사용)

from typing import Literal, List, Optional
from pydantic import BaseModel, model_validator


class ChoiceOption(BaseModel):
    id: str
    label: str
    value: str
    icon_hint: Optional[str] = None
    is_other: bool = False


class QuestionItem(BaseModel):
    id: str
    type: Literal["choice", "narrative"]
    text: str
    target_age: Literal["elderly", "youth", "both"] = "both"
    allow_multiple: bool = False
    max_selections: int = 1
    has_other: bool = False
    options: Optional[List[ChoiceOption]] = None
    guidelines: Optional[List[str]] = None
    placeholder: Optional[str] = None
    min_length_hint: Optional[str] = None
    suggested_duration_seconds: Optional[int] = None  # narrative 전용


class QuestionSet(BaseModel):
    schema_version: Literal["1.0"] = "1.0"
    question_type: Literal["choice", "narrative", "mixed"]
    questions: List[QuestionItem]

    @model_validator(mode="after")
    def validate_mixed_structure(self):
        if self.question_type == "mixed":
            if not self.questions or self.questions[0].type != "choice":
                raise ValueError("mixed 타입의 첫 질문은 반드시 choice여야 합니다.")
            for q in self.questions[1:]:
                if q.type != "narrative":
                    raise ValueError("mixed 타입의 2번째 이후 질문은 반드시 narrative여야 합니다.")
        return self
```

### 4-2. `public_data.py` 수정 — instructor 기반 리팩터링

```python
# backend/app/services/public_data.py (관련 부분)
#
# 질문 생성은 기존과 동일하게 DeepSeek(OpenAI 호환 API)을 사용한다.
# instructor.from_openai() + mode=JSON 으로 스키마를 강제한다.
# (DeepSeek은 tool_use보다 JSON 모드가 더 안정적)

import openai
import instructor
from pathlib import Path
from app.config import settings
from app.services.question_parser import QuestionSet, DEFAULT_FALLBACK

_client = instructor.from_openai(
    openai.AsyncOpenAI(
        api_key=settings.deepseek_api_key,
        base_url="https://api.deepseek.com/v1",
    ),
    mode=instructor.Mode.JSON,
)
PROMPT_DIR = Path(__file__).parent.parent / "prompts"


async def generate_topic_question(
    title: str,
    description: str,
    media_type: str,
    target_age: str,
    question_type: str,        # "choice" | "narrative" | "mixed"
    custom_hint: str = "",
) -> QuestionSet:
    if not settings.deepseek_api_key:
        return DEFAULT_FALLBACK

    template = (PROMPT_DIR / f"topic_publish_{question_type}_v1.txt").read_text(encoding="utf-8")
    prompt = template.format(
        title=title,
        description=description,
        media_type=media_type,
        target_age=target_age,
        custom_hint=custom_hint,
    )
    try:
        return await _client.chat.completions.create(
            model="deepseek-chat",
            response_model=QuestionSet,
            max_retries=2,          # 스키마 불일치 시 자동 재시도
            messages=[{"role": "user", "content": prompt}],
        )
    except Exception:
        return DEFAULT_FALLBACK
```

> **역할 분리 유지**: 대화(스트리밍)는 기존대로 `claude.py`(Claude Sonnet), 질문 생성은 `public_data.py`(DeepSeek) — API 키와 모델을 섞지 않는다.

```python
# 폴백 상수 (question_parser.py 하단에 정의)
DEFAULT_FALLBACK: QuestionSet = QuestionSet(
    question_type="narrative",
    questions=[QuestionItem(
        id="q1", type="narrative",
        text="오늘 어떤 이야기를 나눠 보고 싶으신가요?",
        target_age="both",
        placeholder="자유롭게 이야기해 주세요...",
        guidelines=["오늘 하루 기억에 남는 일", "요즘 관심 있는 것"],
    )]
)
```

### 4-3. API 엔드포인트 (신규/변경)

| 메서드 | 엔드포인트 | 설명 |
|--------|-----------|------|
| `POST` | `/welfare/topics/generate-questions` | AI에게 질문 생성 요청. `question_type`에 맞는 프롬프트 템플릿 로드 |
| `POST` | `/welfare/topics/refine-questions` | **신규** — 기존 QuestionSet + 복지사 수정 지시를 AI에게 전달, 개선된 QuestionSet 반환 |
| `POST` | `/welfare/topics/validate-questions` | 복지사가 수정한 JSON의 Schema 유효성 검증 |
| `POST` | `/welfare/topics/publish` | **기존** — `choices` 필드 포함하여 발행. `question_parser`로 검증 후 저장 |
| `POST` | `/welfare/topics/save-draft` | 임시 저장 (미발행). `topic_proposals`에 `is_draft=true`로 저장 |
| `GET` | `/welfare/surveys/current` | 이번 주 발행된 설문지 조회 |
| `GET` | `/welfare/surveys/history?weeks=4` | 최근 N주간 발행 이력 |
| `POST` | `/welfare/surveys/{id}/clone` | 지난 주 설문지를 이번 주 draft로 복제 |
| `PATCH` | `/welfare/surveys/{id}/stop` | 발행 중지 |
| `GET` | `/welfare/surveys/{id}/edit` | 편집용 데이터 조회 |

### 5-3. `survey_responses` 테이블 신규 — 구조화된 답변 누적

> 기존 `conversations` 테이블은 자유 텍스트 저장 → 집계 분석 불가.  
> `survey_responses`는 질문·보기 단위로 답변을 저장하여 `/analytics` API의 집계 쿼리를 직접 지원한다.

```sql
CREATE TABLE survey_responses (
    id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    topic_id              UUID REFERENCES weekly_topics(id) NOT NULL,
    user_id               UUID NOT NULL,
    question_id           VARCHAR(10) NOT NULL,        -- "q1", "q2"
    question_type         VARCHAR(10) NOT NULL,        -- "choice" | "narrative"

    -- choice형 답변
    selected_option_id    VARCHAR(20),                 -- "opt_1", "opt_other" (구조 연결용)
    selected_option_label VARCHAR(100),                -- "채소·생선 좌판" (답변 시점 label 스냅샷)

    -- narrative형 답변 (음성→STT 변환 결과 또는 텍스트 직접 입력)
    narrative_text        TEXT,

    responded_at          TIMESTAMP DEFAULT NOW(),

    UNIQUE (topic_id, user_id, question_id)
);

-- 보기별 집계 쿼리용 인덱스
CREATE INDEX idx_survey_resp_topic  ON survey_responses(topic_id);
CREATE INDEX idx_survey_resp_option ON survey_responses(topic_id, question_id, selected_option_id);
```

**집계 쿼리 예시**:
```sql
-- 보기별 선택 비율 (weekly_topics JOIN 불필요, 보기 문구 수정 후에도 과거 기록 정확)
SELECT selected_option_label, COUNT(*) AS cnt,
       ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER (), 1) AS pct
FROM survey_responses
WHERE topic_id = $1 AND question_id = 'q1' AND question_type = 'choice'
GROUP BY selected_option_label
ORDER BY cnt DESC;

-- 서술형 답변 전체 (수필 생성 입력으로도 활용)
SELECT narrative_text
FROM survey_responses
WHERE topic_id = $1 AND question_type = 'narrative'
ORDER BY responded_at;
```

9. API 연동: `/welfare/surveys/current`, `/history`, `/clone`, `/edit`, `/stop`
10. `SurveyAnalytics.tsx` 신규 — `/analytics` API 연동, 보기별 비율 차트 + 서술형 답변 목록

### Phase 3: 모바일 앱 — 동적 설문지 위젯 (2일)
1. `hooks/useSurvey.ts` 신규 — `QuestionSet` 파싱, 순차 네비게이션
2. `ElderlyMain.tsx` — `question_type`별 하드코딩 제거, `useSurvey` 기반 동적 렌더링
3. `YouthMain.tsx` — 동적 렌더링 (choice 칩, narrative 입력, mixed 순차)

### Phase 4: 통합 테스트 (2일)
1. **E2E 흐름 테스트**: 주제 검색 → AI 질문 생성 → 복지사 보기 추가/삭제/순서 변경 → 발행 → 모바일 조회 → 답변 → 저장
2. **AI JSON 안전성 테스트**: Markdown 코드 블록 → 정상 파싱 / 잘못된 JSON → 재요청 → 폴백
3. **복지사 편집기 테스트**: 보기 1개만 남기고 삭제 시도 → 삭제 차단 / `is_other` 보기 삭제 → 발행 가능
4. **모바일 mixed 흐름 테스트**: 노인 choice 버튼 → narrative 자동 노출 → 음성 녹음 → 완료

## 10. 테스트 시나리오

| # | 시나리오 | 검증 내용 |
|---|---------|----------|
| 1 | AI 선택형 생성 | `POST /generate-questions` → `options` 4~5개, 마지막 `is_other` 포함 |
| 2 | AI mixed 생성 | `question_type=mixed` → 첫 질문 `choice`, 이후 `narrative`, 분기 없음 |
| 3 | 복지사 보기 추가 | 편집기 `[+ 보기 추가]` → 6번째 보기 추가 → 저장 시 `options` 길이 6 |
| 4 | 복지사 보기 삭제 | 보기 2개 남은 상태에서 삭제 시도 → "최소 2개 필요" 토스트 + 삭제 차단 |
| 5 | 복지사 보기 순서 변경 | 드래그로 1번 보기를 3번으로 이동 → 저장 후 모바일에 반영 |
| 6 | 복지사 문구 수정 | `label` inline 수정 → `PUT /edit` → DB 반영 → 모바일에 반영 |
| 7 | instructor 재시도·폴백 | AI 스키마 불일치 → instructor 자동 재시도(2회) → `InstructorRetryException` → `DEFAULT_FALLBACK` 반환 |
| 8 | 노인 mixed 흐름 | `choice` 버튼 터치 → `goToNext()` → `narrative` 화면 노출 → 음성 녹음 → 전송 |
| 9 | 청년 mixed 흐름 | `choice` 칩 선택 → `answers` 상태 업데이트 → `narrative` 텍스트 입력 → `sendAnswers()` |
| 10 | 기타 보기 처리 | 노인: `is_other` 선택 → 음성 녹음 활성화. 청년: `is_other` 선택 → 텍스트 입력 활성화 |
| 11 | 발행 이력 복제 | 지난 주 설문지 `POST /clone` → 이번 주 `draft` 생성 → 편집기에서 수정 → 발행 |
| 12 | 템플릿 저장·불러오기 | 현재 설문지 `POST /survey-templates` 저장 → 다른 주에 `GET /survey-templates` 불러오기 |
| 13 | 선택형 응답 집계 | 사용자 5명 답변 후 `GET /analytics` → 보기별 count·percent 정확히 계산 |
| 14 | 서술형 답변 누적 | 사용자 3명 서술형 답변 → `survey_responses`에 저장 → `/analytics`의 `sample_responses`에 노출 |
| 15 | AI 협의 — 보기 수정 | `POST /refine-questions` + "보기를 더 쉬운 말로 바꿔줘" → 미리보기 즉시 갱신, DB 저장 안 됨 |
| 16 | AI 협의 — 보기 추가 | "포장마차 보기 추가" 요청 → 응답 QuestionSet에 새 option 포함 → 미리보기 반영 |
| 17 | AI 협의 후 발행 | 협의 완료 → "이대로 발행" → `POST /publish` → `weekly_topics.choices`에 최종 QuestionSet 저장 |
| 18 | AI 협의 후 직접 편집 | 협의 완료 → "직접 수정" → `SurveyEditor`에서 보기 순서 변경 → 저장 → 발행 |
