# 이음(以音) 설문지 프롬프트 '안개비' 재설계 리포트

> 작성일: 2026-06-09
> 대상 파일: `backend/app/prompts/` 내 4개 프롬프트 템플릿
> 작성자: 복지사(사용자) + AI 협의

---

## 1. Executive Summary

현재 시스템의 4개 설문지 생성 프롬프트(choice/narrative/mixed/refine)는 기계적 형식(보기 개수, 글자 수, JSON 스키마)에만 집중하여, **사람의 마음을 어루만지는 대화**가 아닌 **정보 추출용 질문**을 생성하고 있었습니다.

이번 재설계는 다음의 핵심 철학을 프롬프트에 내재화했습니다:
- **"안개비를 맞듯이"**: 사용자에게 강제 공감을 요구하지 않고, 주제의 구체적 장면·소리·냄새를 통해 이야기가 자연스럽게 흐르도록 유도
- **"이야기의 공간"**: 추억을 끌어내는 갈고리가 아니라, 이야기가 머물 수 있는 열린 공간 제공
- **"설문지 초안 틀"**: AI가 만든 것은 완성품이 아닌 뼈대이며, 복지사가 현장의 사람들에게 맞게 직접 문구를 수정하고 발행하는 과정을 전제

## 2. 문제 인식 (Why)

### 2-1. 기존 프롬프트의 구조적 결함

| 구조적 결함 | 현상 | 위험성 |
|---|---|---|
| **정서 직접 추출** | `"그때 기분은 어땠나요"`, `"누구와 함께 갔는지"` | 외로운 사용자에게 '혼자'임을 강요, 상처받은 사용자에게 감정 노출 압박 |
| **최고급 강요** | `"가장 기억에 남는 순간"` | 기억이 나지 않으면 실패감, 대화 거부 유발 |
| **명령형 답변 압박** | `"~해 주세요"`, `"~말씀해 주세요"` | 선택의 자유를 박탈, 의무감 부여 |
| **추상적 보기** | `"행복한 기억"`, `"가족과의 시간"` | 보편적 진부함, 사용자의 구체적 삶과 무관 |
| **'기타' 소외** | `"기타 (직접 말씀해 주세요)"` | 사용자를 '나머지' 카테고리로 분류 |
| **기계적 역할 정의** | `"대화 질문 설계 AI"` | 외로움과 상처를 아는 복지 현장의 맥락 부재 |

### 2-2. 사용자(복지사)의 핵심 지적

> *"현재 시스템의 프롬프트에서 설문을 만들기 위해서는 우리가 무엇 때문에 이를 만드는지에 대한 고뇌가 없어."*

> *"청년은 상처를 가지고 있고, 노년은 외로움이 있어. 왜 이걸 생각 안 해?"*

> *"강제 공감을 할 경우의 부작용이야. 주제를 기반으로 안개비를 맞듯이 조심스럽게 다가가서 이야기를 만들기를 원해."*

> *"출력시 우리는 이를 설문지로 만들기 위한 틀로 약속을 했어. 복지사가 미리보기에서 질문의 내용을 수정할 수 있어야 해."*

## 3. 작업 내용 (What)

### 3-1. 재설계 원칙 도출

사용자와의 협의를 통해 다음 5개 원칙을 확정했습니다:

| 원칙 | 의미 | 반영 위치 |
|---|---|---|
| **역할 재정의** | "질문 생성기" → "이야기 틀 짜는 사람" | `[역할]` 섹션 전면 교체 |
| **안개비 철학** | 주제의 구체적 장면/소리/냄새로 이야기가 자연스럽게 흐르도록 유도 | `[철학]` 신규 섹션 |
| **강제 공감 금지** | 정서 직접 묻기, 명령형, 최고급 강요, 추상적 보기 등 명시적 배제 | `[금지사항]` 신규 섹션 |
| **열린 보기** | '기타' 폐기 → "그 외에 문득 떠오르는 것" 등 열린 표현 사용 | `[질문 설계 원칙]` + 출력 형식 |
| **초안 틀 명시** | AI 출력 = 완성품 아님, 복지사 수정/발행 전 단계임을 명시 | `[역할]`, `[출력 형식]` 전면 |

### 3-2. 수정 대상 파일

| 파일 | 유형 | 주요 변경 내용 |
|---|---|---|
| `topic_publish_choice_v1.txt` | 선택형 | 역할/철학/금지사항/원칙 신규. guideline을 '답변 방향' → '이야기의 향기'로 재정의. '기타' → '그 외에 문득 떠오르는 것'으로 변경 |
| `topic_publish_narrative_v1.txt` | 서술형 | 동일. placeholder를 '쓰라는 압박' → '이야기가 머물 수 있는 초대'로 변경. `"가장 ~한 순간"` 등 최고급 금지 추가 |
| `topic_publish_mixed_v1.txt` | 혼합형 | 동일. choice → narrative 흐름을 "진입 문 → 안개비 같은 공간"으로 재정의 |
| `topic_refine_v1.txt` | AI 협의 | 동일 철학 적용. 복지사 수정 요청 반영 시에도 금지사항 준수 유지 |

### 3-3. 핵심 문구 변경 비교

| 구분 | 변경 전 (기계적) | 변경 후 (안개비) |
|---|---|---|
| **역할** | `당신은 세대공감 플랫폼 '이음(以音)'의 대화 질문 설계 AI입니다` | `당신은 '이음(以音)'의 대화 설계자입니다. 복지사가 검토하고 수정/확정할 수 있는 설문지 초안 틀을 만드는 것이 당신의 임무입니다` |
| **철학** | 없음 | `이야기는 주제에서 시작하지만, 사람의 삶으로 스스로 흘러가야 합니다` |
| **보기** | `기타 (직접 말씀해 주세요)` | `그 외에 문득 떠오르는 것` |
| **질문 예시** | `가장 기억에 남는 순간은?` | `그 시절 시장에 가면 어떤 풍경이 눈에 먼저 아른거리나요?` |
| **guideline** | `누구와 함께 갔는지` | `그때 공기는 어떤 냄새가 났나요` |
| **placeholder** | `그날의 이야기를 들려주세요...` | `문득 떠오르는 대로 적어 주셔도 좋습니다...` |
| **초안 틀 명시** | 없음 | `이 틀은 완성된 설문지가 아니라, 복지사가 현장의 사람들에게 맞게 다듬을 수 있는 뼈대입니다` |

## 4. 결과물 (Result)

### 4-1. 파일 변경 요약

```
backend/app/prompts/
├── topic_publish_choice_v1.txt      ← 재작성 (안개비 철학 적용)
├── topic_publish_narrative_v1.txt     ← 재작성 (안개비 철학 적용)
├── topic_publish_mixed_v1.txt         ← 재작성 (안개비 철학 적용)
└── topic_refine_v1.txt                ← 재작성 (안개비 철학 적용)
```

### 4-2. 기대 효과

| 대상 | 기대 변화 |
|---|---|
| **AI (질문 생성)** | 기계적 보기 채우기 → 주제의 구체적 장면/소리/냄새를 담은 '기억의 문' 생성 |
| **복지사 (미리보기/수정)** | 완성된 설문지를 덤으로 받는 느낌 → 현장에 맞게 문구를 다듬을 수 있는 열린 틀을 받는 느낌 |
| **노인 사용자** | "기억해라"는 압박 → "그때의 공기가 떠오른다면"이라는 초대 |
| **청년 사용자** | 감정 노출 강요 → 자신의 이야기를 머물 수 있는 공간 |

### 4-3. 위험성 감소

| 기존 위험 | 재설계 후 |
|---|---|
| 강제 공감 → 상처 재발/대화 거부 | 구체적 장면 제시 → 사용자가 스스로 문 열기 |
| '기타' 보기 → 소외감 | '열린 창' → 주인공이 될 수 있는 공간 |
| 최고급 강요 → 실패감 | "문득 떠오르는 대로" → 모든 이야기가 유효함 |

## 5. 이후 과제 (Next Steps)

### 5-1. 코드 레벨 반영

현재 프롬프트의 철학은 텍스트 파일에만 담겨 있습니다. 이 철학이 실제 AI 출력에 안정적으로 반영되도록, 다음 기술적 보완이 필요합니다:

| 과제 | 필요성 | 제안 |
|---|---|---|
| **Pydantic 검증 강화** | AI가 금지사항(정서 직접 묻기 등)을 어길 수 있음 | `question_parser.py`에 금지 키워드/패턴 검증 로직 추가 |
| **복지사 수정 UI 연동** | "초안 틀"이라는 개념이 대시보드에 반영되어야 함 | `SurveyRefineChat.tsx` / `SurveyEditor.tsx`에 "AI 초안 → 복지사 수정 → 발행" 흐름 명시 |
| **수필 연결 맥락** | 질문이 수필의 재료가 된다는 흐름을 AI가 이해해야 함 | 수필 생성 프롬프트(`essay.py` service)와 질문 생성 프롬프트의 철학적 일치성 검토 필요 |

### 5-2. 실전 검증

| 검증 항목 | 방법 |
|---|---|
| AI 출력 품질 | 실제 주제(예: "1960년대 남대문 시장")로 generate-questions 호출 → 보기/질문 문구가 '안개비' 철학을 따르는지 수동 검증 |
| 복지사 수정 흐름 | 대시보드에서 AI 초안 수신 → 문구 inline 수정 → 발행 시나리오 테스트 |
| 사용자 반응 | 모바일에서 수정된 설문지로 실제 대화 시뮬레이션 → 강제 공감 느낌이 사라졌는지 확인 |

## 6. 결론

이번 재설계는 기술적 기능 추가가 아닌, **프롬프트의 영혼을 바꾸는 작업**이었습니다. "보기를 몇 개 만들어라"에서 "이 질문이 외로움과 상처를 어루만질 수 있는가"로 질문의 중심을 옮겼습니다.

AI가 생성하는 것은 더 이상 '완성된 설문지'가 아니라, **복지사가 현장의 사람들에게 맞게 다듬어갈 수 있는 열린 틀**입니다. 그 틀 안에서 이야기는 주제에서 시작해, 사람의 삶으로 스스로 흐르기를 기대합니다.

> *"안개비를 맞듯이."*
